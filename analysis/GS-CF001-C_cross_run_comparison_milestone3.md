## Methodology note on source mutation

**Why mutation is detected despite identical complaint IDs:**

The CFPB Consumer Complaint Database is a live, continuously-updated database.
Between the three pipeline runs (~10 seconds apart), approximately 100 of the
100 retrieved records showed changed content hashes. This is expected and
correct behaviour: the CFPB API returns records whose mutable fields (company
response status, company response timestamp, `date_sent_to_company`, and
similar administrative fields) are updated in real time by the CFPB system,
independent of when the underlying complaint was filed.

**What this means for the pipeline:**

This mutation does not affect mechanism classification, proof gate outputs,
verdicts, or ODR entries. The pipeline classifies complaints based on their
`complaint_what_happened` narrative and structured taxonomy fields (product,
issue, sub-issue) — these fields are stable across runs. The mutable fields
that differ are administrative metadata not used in classification.

**What this means methodologically:**

CFPB records are not immutable. The source mutation detection (Correction 4)
correctly surfaces this property of the data source. It confirms that:

1. The `complaint_id` set is stable within a short session (all 100 IDs returned identically).
2. The ordering of results is stable across runs.
3. Some administrative record fields can change between pulls, even within seconds.

This finding should inform downstream data governance: if reproducibility across
days or weeks is required (not just within a session), raw artifacts must be
preserved per run, and cross-run comparisons should note the mutation rate.
